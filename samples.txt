### Client-side log streaming deadlock (VERY common in Prefect 3)
Ad hoc runs often try to stream logs back to the client.
Key signal:
Scheduled runs don’t stream logs to your workstation → they work fine
Test this theory:

Run WITHOUT log streaming:
prefect deployment run <deployment-name> --no-wait

OR in Python:
from prefect.deployments import run_deployment

run_deployment("name", timeout=0)
If this works → log streaming is the culprit
################


If the CLI is too slow or you have millions of rows, you can go directly into Postgres (via PGAdmin or psql) and truncate the logs. This is much faster.

SQL
-- This keeps your flow history but deletes the heavy text logs
TRUNCATE TABLE log;

-- OR, if you want to keep recent logs but kill the bulk:
DELETE FROM log WHERE created < NOW() - INTERVAL '7 days';

-- CRITICAL: After deleting, you must 'reclaim' the space
VACUUM FULL log;
Intuition: Imagine a physical filing cabinet. Even if you throw away the papers, the cabinet still takes up the same space in the room. VACUUM FULL actually shrinks the "cabinet" so the Server can search it faster.

3. The "Automated" Way (Retention Policy)

If you don't want to do this again in three months, you should set a Retention Policy. In Prefect 3.x, you can automate this using a simple maintenance flow that runs once a week.

Python
from prefect import flow
from prefect.client.orchestration import get_client
import datetime

@flow
def cleanup_old_runs():
    async with get_client() as client:
        # Define your 'cutoff'
        before = datetime.datetime.now() - datetime.timedelta(days=30)
        # This is a high-level API call to purge the DB
        await client.delete_flow_runs(before=before)

if __name__ == "__main__":
    cleanup_old_runs()
