Field_Name	Data_Type	Nullable	Definition
Transaction_Key	INT PK	No	Surrogate primary key. Auto increment integer assigned by SQL Server using IDENTITY(1,1). Uniquely identifies each transaction row. Has no business meaning — exists solely to uniquely identify the row.
Date_Key	INT FK	No	Foreign key to Dim_Date. The business date this transaction was posted. Used to scope transactions to a specific period for YTD aggregation in the ETL.
Account_Key	INT FK	No	Foreign key to Dim_Account. The account this transaction occurred against. Used to join through Bridge_Account_Resp to attribute the transaction to the correct responsibility entity.
Product_Key	INT FK	No	Foreign key to Dim_Product. The product type associated with the account this transaction occurred against. Allows filtering and grouping of transaction activity by product category.
Transaction_Amount	DECIMAL(18,4)	No	The dollar amount of the transaction. Positive for credits — deposits, payments, draws. Negative for debits — withdrawals, fees, reversals. Use ABS() when summing volume regardless of direction.
Transaction_Type_Cd	VARCHAR(20)	No	Classifies the type of transaction. Examples: DEPOSIT, WITHDRAWAL, PAYMENT, DRAW, TRANSFER, FEE, PAYOFF, REVERSAL. Used to filter specific activity types for goal metrics.
Channel_Cd	VARCHAR(20)	Yes	The channel through which the transaction was initiated. Examples: BRANCH, MOBILE, ATM, ONLINE, WIRE, ACH. NULL if channel is not captured by the source system. Used for channel mix reporting.
Source_System_Cd	VARCHAR(20)	No	Identifies which source system originated this transaction. Examples: CORE_BANKING, LOAN_SERVICING, PAYMENT_PROC. Combined with Source_Transaction_Id forms the natural key that prevents duplicate loads across ETL runs.
Source_Transaction_Id	VARCHAR(50)	No	The original transaction identifier from the source system. Combined with Source_System_Cd forms a unique natural key. Allows the ETL to detect and skip duplicate rows if the same transaction is sent more than once from the source system.
Load_Date	DATE	No	The date this row was loaded by the ETL process. Used for pipeline auditing and troubleshooting. May differ from Date_Key if transactions are loaded with a delay from the source system.
goal_level_cd           VARCHAR(10)     Indicates the organizational level the
                                        goal is assigned to.
                                        Values: 'RM' or 'BRANCH'
                                        Use to avoid double-counting when mixing
                                        RM-level and branch-level goals in reports.

goal_version_cd         VARCHAR(20)     Identifies which annual plan this goal
                                        came from. Allows multiple years to
                                        coexist in the same table.
                                        Example: 'FY2025_BUDGET', 'FY2024_BUDGET'


── ANNUAL GOAL REFERENCE ───────────────────────────────────────────────────────

Field                   Type            Definition
─────────────────────── ─────────────── ────────────────────────────────────────
annual_goal_amount      DECIMAL(18,2)   The original full-year goal as set by
                                        Finance or management at the start of the
                                        year. This number never changes once the
                                        year begins. Preserved for auditability.

total_biz_days_in_year  INT             Total number of business days in the
                                        fiscal year. Typically 261. Used as the
                                        denominator when prorating the annual goal
                                        to a YTD amount.


── CUMULATIVE YTD MEASURES ─────────────────────────────────────────────────────

Field                   Type            Definition
─────────────────────── ─────────────── ────────────────────────────────────────
biz_days_elapsed        INT             How many business days have passed from
                                        Jan 1 through the snapshot date. Drives
                                        both the prorated YTD goal and the
                                        projected year-end calculation.
                                        Source: dim_date.business_days_elapsed

ytd_goal_amount         DECIMAL(18,2)   The prorated portion of the annual goal
                                        through today's snapshot date.
                                        Formula:
                                          annual_goal_amount
                                          / total_biz_days_in_year
                                          * biz_days_elapsed
                                        Represents "what you should have done
                                        by now if performance were perfectly
                                        linear across the year."

ytd_actual_amount       DECIMAL(18,2)   Cumulative actual performance from Jan 1
                                        through the snapshot date. This is the
                                        sum of all daily actuals for this
                                        RM / product / metric combination pulled
                                        from the source system (LOS, core banking,
                                        etc.).

ytd_variance_amount     DECIMAL(18,2)   Dollar or unit difference between actual
                                        and goal year to date.
                                        Formula: ytd_actual_amount
                                                 - ytd_goal_amount
                                        Positive = ahead of goal
                                        Negative = behind goal

ytd_attainment_pct      DECIMAL(8,4)    Percentage of the prorated YTD goal that
                                        has been achieved. This is the primary
                                        scorecard metric.
                                        Formula: ytd_actual_amount
                                                 / ytd_goal_amount
                                        1.0000 = exactly on pace (100%)
                                        > 1.0  = ahead of goal
                                        < 1.0  = behind goal
                                        Note: always divide the period sums —
                                        never sum the daily attainment_pct values.

projected_year_end      DECIMAL(18,2)   Full-year estimate based on the current
                                        run rate. Answers the question: "If the
                                        RM keeps performing at this pace, what
                                        will they close by Dec 31?"
                                        Formula: (ytd_actual_amount
                                                  / biz_days_elapsed)
                                                 * total_biz_days_in_year


── AUDIT ───────────────────────────────────────────────────────────────────────

Field                   Type            Definition
─────────────────────── ─────────────── ────────────────────────────────────────
load_date               DATE            The date this row was written by the ETL
                                        process. Should match the snapshot date
                                        for daily loads. Used for troubleshooting
                                        and pipeline auditing.

================================================================================
 KEY OBSERVATIONS FROM SAMPLE DATA
================================================================================

1. J. Smith — mixed performance
   HELOC originations are strong at 110.5% but savings balance lags at 85.5%.
   Without tracking both metrics you would only see half the picture.

2. M. Jones — projected year-end gap is the critical number
   Auto loan originations are at 73.5% attainment. Projected year end of
   $2.94M against a $4M goal means a $1.06M gap. Knowing this in May gives
   management time to course correct before Q4.

3. Dollar attainment and account count can diverge
   J. Smith is at 110.5% on HELOC dollars but only 95% on account count —
   his deals are larger than average. M. Jones is the opposite — smaller
   deals, fewer accounts. Both dimensions are needed to manage performance.

4. ytd_attainment_pct is always computed as SUM(actual) / SUM(goal)
   Never average the daily attainment values. Always divide the cumulative
   period totals to get a meaningful attainment percentage.
